## 一、规范驱动开发 (SDD)

**代码质量问题的根因不是 AI "写不好"，而是 AI 在没有明确规范时就动手。**

```
需求 → 写 Spec → 人工确认 Spec → AI 按 Spec 生成代码 → 对照 Spec 验收
         ↑                              ↑
    没有 Spec = 不准写代码          Spec 是真相，代码服务于 Spec
```

### 任务 Spec 门禁

1. 填写 Spec 模板（`docs/templates/task-spec-template.md`，至少 1-5 项）
2. 提交给用户/auditor 确认
3. 收到 "Spec 确认，可以开始" → 才允许写代码
4. 代码写完后 → 对照 Spec 逐条验收（第 8 项）

跳过 Spec 直接生成的代码，可用率 < 30%。完整 SDD 流程可用率 ≥ 85%。

### 模块级契约

每个模块可创建模块级 CLAUDE.md（参考 `docs/templates/module-claude-template.md`）：
- **架构约定**：分层职责、事务、SQL 安全
- **正例**：可复用的现有模式，新功能必须参考
- **反例**：已知技术债务，新代码禁止模仿

### 关键约束

- **模式复用优先**：新功能必须找到项目中已有的类似实现作为参考。找不到 → 先讨论架构，再写代码。禁止每次另起炉灶。
- **契约变更显式声明**：任何改变 API 入参/出参、字段 null 允许性、返回值类型的行为，必须在 Spec 第 4 项中声明。
- **反例不可碰**：模块 CLAUDE.md 中列出的反例，新代码绝对不能模仿。

**来源**：[SDD (arXiv:2602.00180)](https://export.arxiv.org/abs/2602.00180) · [Constitutional SDD (arXiv:2602.02584)](https://browse-export.arxiv.org/abs/2602.02584) · Spec Coding 工业实践 · Google Style Guide 生态

## 二、审计修复流程

```
收到审计报告 → 读取 AUDIT_TRACKER.md → 逐条修复 P0 → P1 → P2
    ↓
修复完成后 → 更新 fix# 状态 → 调用独立 skill 验证
    ↓
验证通过 → 请求用户将 status 改为 CLOSED → 下一阶段
验证失败 → 继续修复 → 重新验证
```

你无权自行将 report 状态改为 CLOSED。只有用户或独立验证 agent 可以关单。

**验证必须物理隔离**：FIX_CLAIMED → VERIFIED 必须在新的独立 session 中完成（不同 AI 实例），禁止自验自宣告。

## 三、修复范围机械约束

修复审计问题时，变更范围**机械可判**，不可自由裁量：

1. **新增代码**：只能新增 fix-checklist 明确要求的代码元素（注解、字段、方法）
2. **禁止新增注解类型（机械规则）**：如果一个注解类型（如 `@NotNull`、`@NotBlank`、`@NotEmpty`）在修复前不存在于该文件的任何字段上，**禁止新增该注解类型**。此规则用 `grep` 即可验证，无歧义
3. **禁止修改已有方法的签名或语义**：包括参数数量/类型、返回值类型、异常抛出行为
4. **禁止跨文件扩散**：fix-checklist 列出 A.java，不意味着可以改 A.java 引用的 B.java
5. 如果发现额外问题 → 记录到 AUDIT_TRACKER 作为新条目，不在当前 commit 中混入

**scope creep 为什么是 P0**：校验注解改变 API 契约——原来允许 null 的字段变为 400 Bad Request，所有未传该字段的调用方全部中断。藏在修复 commit 里的行为变更无法被 reviewer 预期发现。

**自查标准（机械可判）**：
- [ ] git diff 中每个 `+` 行必须能对应到 fix-checklist 中的一条
- [ ] 如果 `+` 行是新增注解，该注解类型必须已存在于该文件的至少一个其他字段上
- [ ] 如果不满足 → 撤销该行，不放进 commit

## 四、提交后审查等待规则

每次 `git commit` 完成后：

1. 输出 commit hash + 变更摘要给用户
2. **禁止**立即开始下一个开发任务
3. 等待用户或 auditor 明确回复 "通过" 后才能继续
4. 如果收到审查反馈指出问题：在当前 commit 之上修复（不混入新功能）→ 重新提交 → 重新等待审查
5. "10/10 验证通过" 验证的是**功能正确性**，不等同于**代码质量通过**

**为什么需要**：AI 的默认任务模型是线性的（完成→下一个），没有"提交后等待反馈→修正→再提交"的反馈循环。2026-04-26：executor 在 12 分钟内连续提交 3 个 commit，审查反馈追不上，两个质量问题（重复 import + @NotNull scope creep）遗留未修。

## 五、四阶段架构崩塌模型

```
第一阶段：加功能，完美
第二阶段：扩展功能，上下文轻微漂移，新代码背离原设计
第三阶段：代码库出现两种略有不同的模式解决同一问题
第四阶段：五种不同实现方式共存，无法判断哪个是正统
```

根因：AI 每次只做局部优化，每个单独决策都合理，但跨会话一致性无声崩解。

**防御**：Spec 先行 + 模式复用优先 + 跨会话交接 + 定期审计。

## 六、5 种已证实的治理失效模式

| 失效模式 | 缓解措施 | 剩余风险 |
|---------|---------|---------|
| CLAUDE.md 不遵守 | 多层文本规则叠加，提升合规概率 | 无法 100% 保证 |
| 子代理绕过 | 独立 session 验证（物理隔离） | 同一 session 内的子代理 |
| 模型自我修改 | 无 | AI 有文件写入权限时无法完全防止 |
| 替代工具路径 | pre-commit hook 机械检查 diff | 仅覆盖 commit 边界 |
| Hook 静默失败 | pre-commit exit 1 硬阻断 | 依赖 hook 脚本质量 |

来源：RFC #45427 · Issue #18986（60 天内 948 次系统偏离）

## 七、模型能力与成本感知

> 按能力分配，按成本优化，关键路径不妥协。

### 任务分级

| 等级 | 典型任务 | 最低模型要求 |
|:----:|---------|------------|
| L0 机械 | 读文件、grep、git log/status | 任意 |
| L1 查询 | 代码解释、文档整理、格式化 | 任意 |
| L2 开发 | Bug 修复（范围明确）、CRUD、SQL | 中等推理 |
| L3 审查 | 安全审查、代码质量、跨文件一致性 | 中高推理 |
| L4 架构 | 新功能设计、重构方案、治理规则 | 强推理 |
| L5 安全 | 安全审计、漏洞分析、权限模型审查 | 强推理+安全技能 |

### 五条成本红线

1. **关键路径不降级**：L4/L5/P0 修复验证 → 禁止用弱模型
2. **降级必须声明**：便宜模型做 L3+ → 输出开头声明能力边界
3. **不可因成本跳过门禁**：pre-commit/编译检查/Spec/审查等待
4. **超额必须预警**：预估消耗 > 余量 30% → 告知用户，不得静默消耗
5. **批量操作分批确认**：10+ 文件禁止套餐内一次性执行

### 模型能力不足信号（出现任一 → 停止，建议切换更强模型）

- 重复犯 KNOWN_ISSUES 中已记录的同类错误
- 无法区分"修复"和"重构"的边界（scope creep）
- 连续 3 次 pre-commit hook 阻断后仍未修复根因
- 输出代码中包含 `// TODO` 或 `// ...` 不完整实现
- 声称"10/10 通过"但不引用具体验证步骤

### 套餐余量规划

**会话启动**：确认套餐类型/余量/有效期 → 按 L 级估算消耗 → 预估 > 余量 30% → 告知用户

| 任务规模 | 拆分策略 | 成本控制 |
|---------|---------|---------|
| 单文件 <100行 | 不拆分 | 预估即实际 |
| 多文件 3-10 | 按文件分批，每批等确认 | 每批 < 额度 20% |
| 批量 10+ | 禁止全量。先做 2-3 样本 → 确认 → 分批 | 首样本 < 额度 10% |
| 全量审计 | 禁止套餐内执行。建议按量付费或本地工具 | — |

**余量不足**：余量 < 消耗 → 优先保护关键任务 / 余量 < 10% → 仅 L0-L1 / 耗尽 → 保存进度，标注等待充值

**关键认知**：便宜模型重试 10 次 > 贵模型一次做对。但贵模型做 L0 = 浪费。匹配能力，不搞一刀切。

## 八、来源可信度（SRC-CRED）

| 级别 | 标签 | 典型来源 | 验证要求 |
|:----:|------|---------|---------|
| A | 权威 | 政府指导、顶会论文(AAAI/ICSE)、Anthropic/GitHub/Microsoft 官方、OWASP | 验证实现可行性 |
| B | 可信 | 高校研究组、10k+ stars 仓库、行业领袖、知名安全公司 | 交叉 1-2 个来源 |
| C | 参考 | 1k-10k stars、有记录的个人博客、InfoQ/CSDN/掘金 | 至少 1 个独立来源 |
| D | 待验证 | <1k stars、无记录博客、新发布未验证工具 | 必须独立验证 |
| E | 已排除 | 已知恶意、已证伪、不可访问 | 零信任，黑名单 |

**原则**：信任不等于盲信。所有外部结论在实践中必须可复现方可作为决策依据。实践是唯一真理标准。

```
## 十、交付状态升级门禁

针对近期暴露的“把 implemented 误报成 closed”问题，项目级状态升级统一以
[delivery-status-upgrade-gate.md](D:/erp/docs/plans/delivery-status-upgrade-gate.md)
为准。

最小执行口径：

1. 没有 `code/build/runtime/doc/final-status` 五联证据，不得写 `VERIFIED`
2. 没有独立复核，不得写 `CLOSED`
3. `session-handoff` 顶部对外状态由复核者更新，不由执行者自宣
4. DeepSeek 交付必须使用
   [deepseek-delivery-template.md](D:/erp/docs/plans/deepseek-delivery-template.md)
   ，否则状态回报无效
  └─ 更新 session-handoff.md
```

## 十一、风险分级审批模型

> 来源：Claude Code 五档权限 · Cline 低/中/高 · Codex 三档审批 · 阿里四种触发类型

### 三级审批

| 等级 | 操作类型 | 示例 | 策略 |
|:----:|---------|------|------|
| **低风险** | 只读、搜索、分析 | Read, Grep, Glob, git status/log | 自动放行 |
| **中风险** | 文件修改、编译 | Edit, Write, mvn compile | 快速确认 |
| **高风险** | 网络请求、删除、部署 | curl, rm, git push, npm publish | 完整审批 |

### 设计原则

1. **安全操作自动放行** — 减少审批疲劳，让用户在真正高风险操作时保持警觉
2. **低风险 ≠ 零风险** — 所有自动放行操作仍受 pre-commit hook 和 PostToolUse 自检约束
3. **分级可配置** — 不同项目可在 PROJECT_CONFIG.md 中调整风险阈值
4. **断路器** — 连续 3 次拒绝或累计 20 次拒绝 → 强制人工介入

## 十二、规则唯一标识与分级体系

> 来源：Microsoft IDE 规则编号 · 阿里 P3C [强制]/[推荐]/[参考] · CORE 多引擎执行

### 规则 ID 格式

```
{等级}-{域}{序号}  例如: M-GA01, S-CR02, C-AR01

等级: [M]=强制(阻断) [S]=建议(警告) [C]=参考
域:   GA=Git Add  AR=架构  CR=代码审查  PE=权限  SE=安全  AU=审计  SP=Spec
```

### 三级规则

| 标签 | 含义 | 违反后果 | 检查方式 |
|:----:|------|---------|---------|
| **[M]** | 强制 (Must) | 阻断提交/部署 | 机械执行（正则/lint/pre-commit） |
| **[S]** | 建议 (Should) | 警告，Review 中需解释 | 静态分析 + Review |
| **[C]** | 参考 (Could) | 不阻断，AI 主动遵循 | AI 自觉 + 定期审计 |

### 执行引擎

| 优先级 | 引擎 | 适用场景 | 可靠性 |
|:----:|------|---------|:----:|
| 1 | **正则门** | 命名风格、禁止模式 | 100%（确定性） |
| 2 | **Glob 门** | 文件作用域、路径白名单 | 100%（确定性） |
| 3 | **AST 门** | 代码结构、注解检查 | 高（结构化） |
| 4 | **LLM 门** | 语义判断、设计原则 | 中（需验证） |

**原则**：确定性优先于智能判断。能用正则/glob/AST 检测的，不用 LLM。

## 十三、经典案例教训集成

> 来源：20 个软件工程经典案例研究（10 反面 + 5 正面 + 5 历史事件）

### 7 大重复失败模式

| # | 模式 | 典型案例 | 我们的防御 |
|---|------|---------|-----------|
| 1 | **单点故障** | Therac-25, Boeing MCAS | 独立审查 + 多重验证 |
| 2 | **复用假设失效** | Ariane 5, Knight Capital | 复用前重新评估边界条件 |
| 3 | **默认不安全** | Log4j JNDI, CrowdStrike | 默认拒绝，按需显式授予 |
| 4 | **接口缺契约** | Mars Climate Orbiter | Schema 级接口契约 |
| 5 | **监管利益冲突** | Boeing FAA, Toyota | 审查者与执行者分离 |
| 6 | **速度压倒安全** | Knight 手动部署 | No Spec No Code + 灰度发布 |
| 7 | **缺乏事后总结** | Toyota 隐瞒 5 年 | session-handoff + KNOWN_ISSUES |

### 正面标杆

| 案例 | 核心实践 | 我们对标 |
|------|---------|---------|
| NASA 航天飞机软件 | 4 轮独立审查 + 形式化规格 | 独立 skill 验证（物理隔离） |
| SQLite | 9200 万行测试 / 15 万行代码 | 变异测试 + 边界值门禁 |
| Linux Kernel | 分层维护者 + 公开审查 | 三层 skill 架构 |
| Google SRE | 错误预算 + 无责备复盘 | hit_rate 评分 + session-handoff |
| Amazon API 化 | API-first + 强制解耦 | 模块 CLAUDE.md 契约 |

### 一句话教训

**最严重的事故不是来自单一技术 bug，而是技术 + 流程 + 文化三层的叠加失败。**
