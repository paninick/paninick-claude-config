# ERP Agent 协作规则

## 核心原则

1. **用户是最终决策者**：所有 Tier 1 决策需要用户确认
2. **垂直委派**：Tier 1 → Tier 2 → Tier 3，不跨层级直接委派
3. **协作实现者**：任何 agent 都不是自主代码生成器，写文件前必须获得用户批准
4. **透明度**：所有架构决策、权衡和偏差都必须明确说明

## 决策工作流

### 战略决策（Tier 1 Agent）

```
1. 理解完整上下文（读取相关文档）
2. 框架化决策（核心问题是什么？）
3. 呈现 2-3 个选项（每个选项的优劣权衡）
4. 给出明确建议（推荐哪个及原因）
5. 等待用户确认
6. 执行并记录决策
```

### 实现工作流（Tier 2/3 Agent）

```
1. 读取 Spec 或 Story
2. 识别模糊点，提问澄清
3. 提出架构方案（展示类结构、文件组织）
4. 解释权衡（简单 vs 灵活、快速 vs 可维护）
5. 询问批准："我可以写入 [filepath] 吗？"
6. 实现并验证
7. 提供后续步骤建议
```

## 冲突升级规则

| 冲突类型 | 升级到 |
|---------|--------|
| 技术方案冲突（两种实现方式都可行） | `erp-cto` |
| 业务规则冲突（业务逻辑不清晰） | `erp-product-owner` |
| 进度/优先级冲突（多个任务竞争资源） | `erp-pm` |
| 安全 vs 功能冲突 | `erp-cto` + 用户确认 |

## 协作协议

### Agent 间通信

当一个 agent 需要另一个 agent 的输出时：
1. 明确说明需要什么（"我需要 erp-lead-db 设计 erp_plan 表结构"）
2. 提供足够上下文（业务需求、约束条件）
3. 等待输出后再继续

### 文件写入规则

- **任何 agent 写文件前必须明确询问用户批准**
- 多文件变更时，列出所有受影响的文件
- 对现有文件的修改，先展示 diff 再询问

### 代码审查规则

- Tier 3 实现的代码必须经过 Tier 2 审查
- 安全相关代码（@PreAuthorize、SQL、JWT）必须经过 `erp-security` 审查
- 数据库变更必须经过 `erp-lead-db` 审查

## 模型使用规则

| 场景 | 模型 | 原因 |
|------|------|------|
| 战略决策、架构评估 | Opus | 需要深度推理和权衡分析 |
| 代码实现、代码审查 | Sonnet | 平衡质量和速度 |
| 快速查询、状态快照、简单生成 | Haiku | 节省成本，速度快 |

## 禁止行为

- 任何 agent 不得修改 `ruoyi-framework/` 目录
- 任何 agent 不得在未获批准的情况下写入文件
- 任何 agent 不得跳过 10 步检查点中的任何步骤
- 任何 agent 不得在 Mapper XML 中使用 `${}` 拼接用户输入
